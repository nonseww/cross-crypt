# cross-crypt

Цель презентации — показать продукт от бизнес-идеи до технической реализации.

Все разделы должны описывать один и тот же сервис и логически продолжать друг друга. То есть между ними должна прослеживаться логика

## Требования к оформлению:

- Все диаграммы должны быть читаемыми.
- Использовать единые названия сущностей во всех артефактах.
- Термины в BPMN, Use Case, ER и API должны совпадать.
- Не использовать разные названия для одной и той же сущности.
- Название слайда совпадает с названием пункта.
- Минимум текста, только указанные необходимые критерии. Без воды.
- Все артефакты должны ссылаться на одни и те же сущности и сценарии.
- Каждый Use Case должен прослеживаться в BPMN, API и UI.
- Основной пользовательский сценарий должен присутствовать минимум в User Stories, Use Case, BPMN и Sequence Diagram.

## Структура презентации

### 1. Название продукта + краткий питч [1 слайд]

Формат:

_Что это за продукт и какую проблему решает_

`1–2 предложения`

### 2. Концепция продукта [1 слайд]

Кратко описать:

- что представляет собой продукт
- как проходит игровой цикл
- чем продукт отличается от аналогов

### 3. Проблема (Business Problem) [1 слайд]

_Зачем это нужно_

- какую проблему/задачу решает продукт
- почему это важно для пользователей

`несколько предложений`

### 4. Бизнес-цели [1 слайд]

_Что хочет бизнес_

`3-4 пункта (монетизация, вовлеченность и так далее). Использовать конкретные метрики и цифры`

### 5. Пользователи [1 слайд]

_Кто использует систему_

```
Определить основные роли системы.

Для каждой роли указать:
- цель использования продукта;
- основные действия;
- ограничения доступа (если есть).
```

`несколько пунктов`

### 6. User Stories (из БТ) [1 слайд]

_Что хочет пользователь_

`Не менее 3-5 User Stories для основных сценариев системы. Вид: "Я [роль] хочу [действие/возможность] чтобы [ценность/цель]"`

### 7. Бизнес-правила [1 слайд]

_Ограничения мира работы нашего приложения_

```4-5 правил, влияющих на логику работы системы.
Не включать сюда нефункциональные требования (система должна отвечать за 2 секунды и так далее - это НЕ бизнес-правило!).
```

### 8. Общая логика системы (high-level architecture) [1 слайд]

- **клиент** (frontend)
- **сервер** (backend)
- **real-time** слой (WebSocket)
- **внешние сервисы** (discord)

```
Диаграмма высокого уровня:
Frontend → Backend → База данных → Внешние сервисы.

Достаточно показать основные компоненты и связи между ними.
```

### 9. Use Cases + UML diagram [1-2 слайда]

_Как пользователь взаимодействует с системой_

- основные сценарии
- include/extend

`use cases на одной части слайда, UML на другой, деление вертикальное. Если не помещается, переносите на следующий слайд`

### 10. Бизнес-процесс (BPMN) [2-4 слайда]

_Как проходит процесс в жизни_

```
Вставить BPMN. Описать основной бизнес-процесс пользователя от входа в систему до достижения результата.
Разбейте рисунок на несколько логических блоков, вы должны быть в состоянии прочесть шрифт.
Выносите данные блоки либо два рядом на одной презентации, если делятся пополам, либо на следующий слайд
```

### 11. Прототип интерфейса (Figma) [1-3 слайда]

```
Показать основные экраны приложения.
Не вставлять все макеты подряд.
Выберите 3-5 самых важных экранов и продемонстрируйте основной пользовательский путь.
На слайде обязательно должна присутствовать ссылка на Figma.
```

### 12. UX. Пользовательские требования [1 слайд]

_Оформлять как нефункциональные требования!_

Пример:

- Пользователь должен получать обратную связь не позднее 100 мс после действия.
- При ошибке должен отображаться визуальный индикатор.
- Игровой интерфейс должен оставаться читаемым даже при наложении помех.

### 13. Модель данных (ER-diagram) [2 слайда]

_Какие сущности есть и связи между ними_

```
На первом слайде текстом расписать про сущности и их роли, на втором слайде вставить ER-diagram.
Показывать только логическую модель.
Типы данных и SQL не указывать.
```

### 14. API (Swagger-like описание)

_Как UI общается с backend_

```
Показать 3-5 ключевых методов API.
Для каждого метода должны быть видны:
- endpoint
- HTTP метод
- параметры запроса
- пример ответа
```

### 15. UML Sequence Diagram [3-4 слайда]

_Как запрос проходит через систему_

```
Вставить 3 UML диаграммы, одна на один слайд.
Если слишком мелко и неичтаемо, разрезайте пополам и вставляйте две части рядом друг с другом на одном слайде.
```

### 16. Интеграции [2 слайда]

_Интеграция с внешним сервисом_

```
На первом слайде кратко написать, как мы интегрируемся в дискорд, почему было принято такое решение и какие его функции/методы используем.
На втором слайде вставить UML Sequence Diagram для интеграции.
```

### 17. Data Flow Diagram [1 слайд]

Показать движение данных между:

- игроком
- клиентом
- сервером
- базой данных
- Discord

### 18. Итоги проекта [1 слайд]

Кратко ответить:

- какую проблему решает продукт
- чем отличается от аналогов
- какие технологии используются
- какие перспективы развития есть у проекта
